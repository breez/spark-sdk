Migration Guide

Breez SDK API Modernization

Background

While Danny was working on the new SDK page, he asked if we could demonstrate how easy it is to integrate the SDK with just a few lines of code. So we put together a minimal connect-and-pay snippet. As we iterated, we kept simplifying it by removing configuration and implementation details.

SDK page snippet (old API)
import { connect, defaultConfig } from '@breeztech/breez-sdk-spark'

const config = defaultConfig('mainnet')
config.apiKey = YOUR_API_KEY

const sdk = await connect({ config, seed, storageDir: './.data' })

const prep = await sdk.prepareSendPayment({
  paymentRequest: 'lnbc50u1p...'
})
await sdk.sendPayment({ prepareResponse: prep })
SDK page snippet (old API): Cleaned-up version
import { connect } from '@breeztech/breez-sdk-spark'

const sdk = await connect({ config, seed })

const prep = await sdk.prepareSendPayment({
  paymentRequest: 'lnbc50u1p...'
})
await sdk.sendPayment({ prepareResponse: prep })

Even after the clean-up, the code was subjectively not very self-explanatory & hard to read in plain English. Which raised two questions:

  • How can we make our API's more self-explanatory & read like natural English?
  • How can we make our actual onboarding logic match the cleaned-up version?

So I looked into how other SDKs with similar concerns to ours address these questions. Some offload configuration to a web dashboard or CLI tool, both not feasible in the short term. Others go with set-and-forget defaults, which would work great for single API key & network setups, but doesn't fully apply to users that have multi-client (wallet) setups with different networks, storage directories, and API key configurations.

Still, there were useful takeaways:

  • Make the simple path genuinely simple
  • Separate what configuration is shared across clients (API key, network) from what's unique to each one (seed, storage)
  • Keep advanced options accessible, but only surface them when needed
Simple path (new API)
import { Breez } from '@breeztech/breez-sdk-spark'

const client = await Breez.connect({ apiKey, network, seed })

const payment = await client.preparePayment('lnbc50u1p...')
await payment.send()
Advanced (new API)
import { Breez } from '@breeztech/breez-sdk-spark'

// Point-of-sale app
const merchant = await Breez.connect({
  apiKey: merchantKey, network: 'mainnet', seed: merchantSeed
})

// Consumer app
const customer = await Breez.connect({
  apiKey: customerKey, network: 'mainnet', seed: customerSeed
})

// Merchant creates invoice, customer pays
const invoice = await merchant.receive({ amountSats: 1000 })
const payment = await customer.preparePayment(invoice.destination)
await payment.send()

Also worth noting: none of the SDKs I looked at used an "SDK" suffix. They use their brand name as the main entry point. Instances are typically suffixed with App or Client (e.g. FirebaseApp, StripeClient). Our old API just exported connect() as a standalone function, with no namespace.

The old API already supported multiple clients, but each connect() call needed its own full config object. Breez.connect() takes a single ClientConfig with everything needed. For advanced customization, use Breez.builder(clientConfig) to get an SdkBuilder for overriding storage, chain service, etc.

Advanced: Builder pattern TypeScript 
// Simple path — one call does everything
const client = await Breez.connect({
  apiKey: 'brz_...', network: 'mainnet', seed
})

// Advanced path — override individual components
const builder = await Breez.builder({
  apiKey: 'brz_...', network: 'mainnet', seed
})

builder.withStorage(myCustomStorage)        // custom persistence
builder.withChainService(myChainProvider)   // custom blockchain provider
builder.withFiatService(myFiatProvider)     // custom fiat rates
builder.withLnurlClient(myHttpClient)       // custom LNURL HTTP client
builder.withPaymentObserver(myObserver)     // payment lifecycle hooks

const client = await builder.build()
Progressive disclosure

Breez.connect(clientConfig) for simple cases. Breez.builder(clientConfig) for advanced customization (custom storage, chain service, etc.).

Resource-oriented design

Operations grouped by what they operate on: client.payments, client.events, client.lightningAddress. Type client. and your IDE shows a handful of resource groups, not 30+ flat methods. Each group has only the methods that make sense for it.

Idiomatic naming

New API: client.preparePayment({destination}) then payment.send(). Reads as: "prepare a payment, then send it."

Old API: sdk.prepareSendPayment({destination}) then sdk.sendPayment({prepareResponse}). Reads as: "prepare a payment, then send the prepared payment." The object already knows it's prepared, you shouldn't have to re-state it.

Hard to misuse

amountSats and feeTokenUnits instead of amount: u128. Prepare, inspect fees, then send. No room for denomination bugs.

7-line rule

A developer should be able to go from zero to a working payment in about 7 lines of code: import the SDK, connect, prepare a payment, and send it. The fewer lines to first success, the faster someone can evaluate whether the SDK fits their needs.

That's where this effort started, and the result is this modernization. It's a strict superset of the existing API. Every deprecated method keeps working, so nothing breaks.

Why Migrate?

The modern API is a strict superset — all deprecated methods keep working. Migrate at your own pace for these benefits:

IDE discoverability

client.payments. autocompletes 2–3 methods, not 30+

No parameter noise

Drop undefined on every optional field — pass only what you need

Unified LNURL

3 calls → 1: preparePayment(destination) accepts string or parsed input

Inspect-then-send

See fees before committing with PaymentIntent

Cross-language consistency

Same names (BreezClient, .send()) in all 9 supported languages

Non-breaking

All deprecated methods still work — migrate at your pace

Entry Point & Connection

Replace defaultConfig() + connect() with a single Breez.connect() call. The BreezSdk type becomes BreezClient.

Problem

defaultConfig() creates a throwaway config every time. No way to share platform config across multiple clients. Storage directory must be manually specified. Invalid API keys only fail at connection time, not at configuration.

Solution

Breez.connect(clientConfig) validates configuration, resolves defaults, and initializes the client in one call. ClientConfig holds all fields (API key, network, seed, storage, etc.). For advanced customization, Breez.builder(clientConfig) returns an SdkBuilder for overriding storage, chain service, etc.

Benefits

Deterministic storage directories keyed to seed fingerprint. Fail-fast validation at creation time. One-call convenience for the common case. Builder pattern for advanced customization.

Send Payment — Unified PaymentIntent Flow

The two-step prepareSendPayment() + sendPayment() becomes preparePayment() returning a PaymentIntent you inspect and send().

Problem

prepareSendPayment() + sendPayment() required different request types per payment method. LNURL needed a completely separate 3-step flow. Fee amounts used a generic u128 — ambiguous between sats and token units. All optional fields had to be passed as undefined.

Solution

Single preparePayment(destination, options) accepts a string or pre-parsed InputType, auto-parses any destination (invoice, Spark address, LNURL, BIP-21). Returns a PaymentIntent with typed fee breakdown, separate amountSats and amountTokenUnits accessors. Call .send() when ready. For LNURL, pass a pre-parsed InputType to avoid a redundant network round-trip.

Benefits

One entry point for all payment types. Typed fees prevent denomination confusion. Prepare–inspect–send flow prevents accidental fee approval. Only pass fields you need.

Receive Payment — Flat Options

Replace the nested paymentMethod discriminated union with flat paymentType + top-level fields.

Problem

ReceivePaymentMethod was a nested discriminated union — callers constructed variant-specific sub-objects ({ type: 'bolt11Invoice', description, amountSats, expirySecs }). Fields scattered across variant arms. The same amount field meant sats or token units depending on context.

Solution

Flat ReceiveOptions with paymentType discriminator and top-level fields. amountSats and amountTokenUnits are separate, explicit fields. Returns ReceiveResult with typed feeSats and feeTokenUnits.

Benefits

No nested construction — just set the fields you need. Denomination is always unambiguous from the field name. Consistent response shape across all payment types.

LNURL-Pay — Unified via preparePayment

No more parse() + prepareLnurlPay() + lnurlPay(). Pass the LNURL directly to preparePayment() — or parse first to discover min/max sendable, then pass the parsed result to skip the redundant network round-trip.

Problem

LNURL-Pay required 3 manual steps: parse() → check result type → extract payRequest → prepareLnurlPay() → lnurlPay(). Callers had to understand LNURL protocol internals and handle type-checking between steps.

Solution

preparePayment(destination, options) accepts either a raw string or a pre-parsed InputType. LNURL-specific options (comment, validateSuccessActionUrl) go in the lnurl sub-object. When you need to inspect minSendable/maxSendable, call parse() first, then pass the result to preparePayment() to skip the redundant network round-trip.

Benefits

3 calls → 1 or 2. No manual type-checking or payRequest extraction. For simple flows, just pass a string. For LNURL flows that need min/max discovery, parse() + preparePayment(parsedResult) avoids double network calls.

New Unified APIs — Send & Receive

Multiple legacy methods are consolidated into two modern entry points with self-documenting fields, typed fees, and a prepare–inspect–send flow.

Problem

Sending required navigating between prepareSendPayment, sendPayment, prepareLnurlPay, lnurlPay — four methods with different request shapes. Receiving required constructing nested discriminated unions with variant-specific fields.

Solution

Two universal entry points: preparePayment(destination, options) for sending (any destination, auto-detected) and receive(options) for receiving (flat options with paymentType discriminator).

Benefits

6+ specialized methods → 2 universal ones. Self-documenting amountSats / amountTokenUnits field names. Same prepare–inspect–send pattern for every payment type.

Replaced by preparePayment()

New unified send

Accepts any destination (invoice, spark address, LNURL, lightning address, BIP-21). Returns a PaymentIntent you can inspect before sending.

Replaced by receive()

Key Differences
1 Self-documenting amounts — amountSats for Bitcoin, amountTokenUnits for tokens (no ambiguous amount)
2 Typed fees — payment.fee.type === 'sparkSats' or 'sparkToken' (no runtime guessing)
3 Prepare–inspect–send — always review fees before confirming; no auto-approve one-liners
4 Any destination — preparePayment auto-detects invoices, spark addresses, LNURL, lightning addresses, BIP-21

New Types & Filtered Events

Key new types, enums, and the filtered event subscription API introduced with the modernization.

Problem

Event listeners received all events and filtered manually (if (event.type === ...)). Amount fields were an ambiguous u128. Configuration mixed platform and client concerns in one flat struct.

Solution

events.on(type, callback) fires only for the requested event type. Typed enums: PaymentIntentType, ReceivePaymentType, PreparedPaymentFee. Single ClientConfig struct for all configuration.

Benefits

Less boilerplate — subscribe to just "payment" events. Fee inspection is type-safe. Single ClientConfig with sensible defaults.

Filtered Event Subscription

Use client.events.on(type, callback) to subscribe to a specific event type instead of filtering manually. Returns a listener ID for .remove().

Before (manual filtering)

  if (event.type === 'paymentSucceeded') {
    handlePayment(event.payment)
  }
})
After (filtered)

const id = await client.events.on('payment', (event) => {
  handlePayment(event.payment)
})

// Clean up
await client.events.remove(id)

Standalone Functions

Some operations no longer require a BreezClient instance. Import and call them directly.

Problem

parse() and checkMessage() required a BreezClient instance even though they use no client state. This forced callers to connect a client before parsing an input or verifying a signature — unnecessary coupling.

Solution

Standalone parseInput() and verifyMessage() imported from the package root. Pure operations with no client dependency.

Benefits

Parse inputs before connecting. Verify signatures without a client. Cleaner API surface — explicit about what needs client state.

Before (required BreezSdk) 
// Parsing required an sdk instance
const parsed = await sdk.parse(input)

// Message verification required an sdk instance
const result = await sdk.checkMessage({
  message, pubkey, signature
})
// Parsing required an sdk instance
let parsed = sdk.parse(input).await?;

// Message verification required an sdk instance
let result = sdk.check_message(CheckMessageRequest {
    message, pubkey, signature
})?;
// Parsing required an sdk instance
let parsed = try await sdk.parse(input: input)

// Message verification required an sdk instance
let result = try await sdk.checkMessage(
    request: CheckMessageRequest(
        message: message, pubkey: pubkey,
        signature: signature))
// Parsing required an sdk instance
val parsed = sdk.parse(input)

// Message verification required an sdk instance
val result = sdk.checkMessage(
    CheckMessageRequest(message, pubkey, signature))
After (standalone) 
import { parseInput, verifyMessage }
  from '@breeztech/breez-sdk-spark'

// No BreezClient instance needed
const parsed = await parseInput(input)

// Pure cryptographic verification
const result = verifyMessage({
  message, pubkey, signature
})
use breez_sdk_spark::{parse_input, verify_message};

// No BreezClient instance needed
let parsed = parse_input(input, None).await?;

// Pure cryptographic verification
let result = verify_message(CheckMessageRequest {
    message, pubkey, signature
})?;
import BreezSdkSpark

// No BreezClient instance needed

// Pure cryptographic verification

    request: CheckMessageRequest(
        message: message, pubkey: pubkey,

// No BreezClient instance needed

1 Import parseInput and verifyMessage from the package root
2 sdk.parse(input) → parseInput(input) — remove BreezClient parameter from function signature
3 sdk.checkMessage({...}) → verifyMessage({...}) — remove BreezClient parameter from function signature

Events — Enriched Synced, Identity Pubkey & LeafOptimization Rename

Three event-related changes: the Synced event now carries granular flags, a new synchronous pubkey getter replaces the need to call getInfo(), and Optimization is renamed to LeafOptimization.

Problem

(a) The SDK emits Synced only after full synchronization, even though balance and transaction data are available earlier. On first restores with large transaction history, this means apps have to wait minutes before showing a balance, even though the data is already there. (b) Identity pubkey required an async getInfo() call even though it’s static per client. (c) Optimization was a vague event name.

Solution

(a) Enriched Synced with balanceUpdated, paymentsUpdated, initialSync boolean flags. (b) client.pubkey synchronous getter. (c) Renamed to LeafOptimization to specify what’s being optimized.

Benefits

(a) Targeted UI refreshes: show balance as soon as it's available, load transaction history in the background. Dismiss loading spinner on initialSync instead of waiting for full sync. (b) No async call for static data. (c) Self-documenting event name.

Enriched Synced Event

The Synced event now carries boolean flags (balanceUpdated, paymentsUpdated, initialSync). This lets you show the balance as soon as it's available and load history in the background, instead of waiting for full sync before rendering anything.

Before (no detail) 
case 'synced':
  // No details — refresh everything
  refreshBalance()
  refreshPayments()
  break
SdkEvent::Synced => {
    // No details — refresh everything
    refresh_balance();
    refresh_payments();
}
case .synced:
    // No details — refresh everything
    refreshBalance()
    refreshPayments()
    break
is SdkEvent.Synced -> {
    // No details — refresh everything
    refreshBalance()
    refreshPayments()
}
After (enriched Synced) 
case 'synced': {
  const { balanceUpdated, paymentsUpdated, initialSync } = event

  if (balanceUpdated)  refreshBalance()
  if (paymentsUpdated) refreshPayments()
  if (initialSync)     dismissLoadingUI()
  break
}
SdkEvent::Synced => {
    // Data has been synchronized with the network.
    // Refresh the payment list and balance.
}
case .synced:
    // Data has been synchronized with the network.
    // Refresh the payment list and balance.
    break
is SdkEvent.Synced -> {
    // Data has been synchronized with the network.
    // Refresh the payment list and balance.
}

Identity Pubkey — Synchronous Getter

Previously you needed getInfo() to retrieve the client's identity public key. Now it's a synchronous property on the client — no async call or sync required.

Before 
// Required an async call + sync
const info = await sdk.getInfo({ ensureSynced: true })
const pubkey = info.identityPubkey
// Required an async call + sync
let info = sdk.get_info(GetInfoRequest {
    ensure_synced: Some(true),
}).await?;
let pubkey = &info.identity_pubkey;
// Required an async call + sync
let info = try await sdk.getInfo(
    request: GetInfoRequest(ensureSynced: true))
let pubkey = info.identityPubkey
// Required an async call + sync
val info = sdk.getInfo(
    GetInfoRequest(ensureSynced = true))
val pubkey = info.identityPubkey
After
// Synchronous — no network call needed
const pubkey = client.pubkey

LeafOptimization Rename

SdkEvent::Optimization is now SdkEvent::LeafOptimization for clarity. The OptimizationEvent type becomes LeafOptimizationEvent.

Before 
case 'optimization': {
  const evt = event.optimizationEvent
  break
}
SdkEvent::Optimization { optimization_event } => {
    // An optimization event occurred
}
case .optimization(let optimizationEvent):
    // An optimization event occurred
    break

    // An optimization event occurred
}
After 
case 'leafOptimization': {
  const evt = event.leafOptimizationEvent
  break
}
SdkEvent::LeafOptimization { leaf_optimization_event } => {
    // A leaf optimization event occurred
}
case .leafOptimization(let leafOptimizationEvent):
    // A leaf optimization event occurred
    break

    // A leaf optimization event occurred
}

1 Synced event: Remove separate 'balanceSynced' / 'historySynced' handlers. Use event.balanceUpdated, event.paymentsUpdated, event.initialSync flags instead

3 Optimization rename: 'optimization' → 'leafOptimization', event.optimizationEvent → event.leafOptimizationEvent, OptimizationEvent → LeafOptimizationEvent
